Definitions — Dorothy


'User Story' and 'Business Story' Share Identical Format — Distinction Undefined
Dorothy · Definitions Analyst
Critical D — Definitions Confidence 10

The skill accepts 'user story' as an input type and produces 'business story' as an output artefact — both using the identical As a... I want... So that... syntax. No distinction is made between them anywhere in the specification.

→ Formally define 'user story' (input) and 'business story' (output). Specify what transformations elevate a user story to a business story — at minimum, a business story should include at least one Given/When/Then scenario.
Four Outcome Types Listed but Not Defined
Dorothy · Definitions Analyst (confirmed by Alexa)
Critical D — Definitions Confidence 10

Olivia is instructed to classify outcomes as Output | State Change | Communication | Null Outcome but none of these types is defined. The Output vs Communication boundary is particularly unclear — a report emailed to a user could be either.

→ Provide a one-sentence definition and one example for each of the four outcome types. Add a disambiguation note for the Output vs Communication boundary.
Severity Mechanically Derived from Confidence — Two Orthogonal Concepts Conflated
Dorothy · Definitions Analyst (amplified by Paul, confidence 10)
Critical D — Definitions Confidence 9

Severity is defined as Critical=8–10, Major=4–7, Minor=1–3 using the same numeric range as the confidence score. Since the confidence filter discards findings below 7, Minor severity findings can never appear and most Major findings (confidence 4–6) are silently discarded. The entire severity taxonomy is broken.

→ Decouple severity from confidence. Define severity independently based on business impact (e.g., "would omission prevent correct implementation?"). Retain confidence as the analyst's certainty signal.
'Feature' Is Central to the Methodology but Never Defined
Dorothy · Definitions Analyst (confirmed by Alexa)
Critical D — Definitions Confidence 9

'Feature' is the primary output unit of Flo's analysis and the basis for all business stories. No definition, size heuristic, or granularity guidance is provided. Two analysts applying the requirements could identify wildly different feature counts from the same document.

→ Define 'feature': a distinct, user-observable capability delivering business value, described by exactly one business story and one or more Given/When/Then scenarios.

'Agent' and 'Analyst' Used Interchangeably Without Definition
Dorothy · Definitions Analyst (confirmed by Alexa)
Major D — Definitions Confidence 9

In Claude Code mode 'agent' means a spawned subprocess; in Claude.ai mode the same entities run sequentially inline and cannot be 'agents' in any technical sense. Using both terms interchangeably obscures this critical architectural distinction.

 Prediction — Paul

Pre-commit Hook Exit Code Contract Entirely Undefined
Paul · Prediction Analyst (confirmed by Olivia, Sophia)
Critical P — Prediction Confidence 10

The entire pre-commit hook behaviour is described in four words: "Pre-commit hook: exit code based." No exit code values, severity thresholds, output format, or relationship to Pipeline mode are specified.

Realistic: Exit 1 on Critical findings, exit 0 otherwise — matching pre-commit hook conventions.
Absurd: Exit code equals the total number of findings found, causing commits to always fail once any requirements exist.
→ Define the full exit code contract: pass condition (e.g., no Critical findings → exit 0), block condition with severity threshold explicitly stated, output behaviour, and relationship to Pipeline mode.
Deduplication Identity Criteria Produce Unpredictable Merging
Paul · Prediction Analyst (confirmed by Alexa confidence 10, Sophia confidence 9)
Critical P — Prediction Confidence 9

Dorothy flags "term X is undefined" (confidence 8). Alexa flags "term X is vague" (confidence 9). Are these the same issue? The "keep highest confidence" rule has no identity criteria. Deduplication is non-deterministic.

Realistic: Aggregator uses semantic similarity, keeps Alexa's finding, discards Dorothy's specific "undefined_term" classification.
Absurd: All findings mentioning the same word token are merged into a single finding attributed to the analyst with the alphabetically first name.
→ Define deduplication identity: two findings are the same issue if they reference the same text excerpt AND share the same finding_type. Different finding types on the same term are distinct findings.

Proposed Glossary

DeFOSPAM
7-step mnemonic framework: Definitions, Features, Outcomes, Scenarios, Predictions, Ambiguity, Missing. Acronym never explicitly decoded in document.
⚠ Acronym undecoded
Feature
A distinct, user-observable capability delivering business value, described by exactly one business story and one or more Given/When/Then scenarios.
⚠ Never formally defined
Finding
A reported issue, gap, or observation produced by an analyst agent, characterised by confidence, severity, finding_type, and recommendation.
⚠ Never formally defined
Analyst
The logical role/persona examining requirements from one DeFOSPAM perspective. Distinct from 'Agent' (execution unit). Conflated throughout document.
⚠ Conflated with Agent
Business Story
Output artefact: "As a [role] I want [action] So that [benefit]" enriched with Given/When/Then scenarios. Identical syntax to User Story (input).
⚠ Input/output distinction absent
Severity
Should measure business impact independently of confidence. Currently incorrectly defined as a direct function of confidence score ranges.
⚠ Definition is incorrect
Confidence Level
Numeric score 1–10 for analyst certainty. Only scores >= 7 are reported. Values 1–6 are suppressed but undefined.
✓ Verified
Outcome
System result classified as Output | State Change | Communication | Null Outcome. All four types are listed but none is defined.
⚠ Types undefined
CRUD
Create, Read, Update, Delete — fundamental data lifecycle operations. Used without expansion.
⚠ Acronym never expanded
NFR
Non-Functional Requirement — quality attributes such as performance, security, scalability. Used without expansion.
⚠ Abbreviation never expanded
Deduplication
Keep highest-confidence finding when multiple analysts flag the same issue. "Same issue" is never defined, making this non-deterministic.
⚠ Identity criteria missing
Provocation Technique
Paul's method: propose one realistic and one deliberately absurd alternative outcome to force stakeholder clarification when an outcome cannot be predicted.
