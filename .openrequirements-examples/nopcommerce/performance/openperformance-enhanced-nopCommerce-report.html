Critical Availability Gaps

NFR-AVL-004: Single Point of Failure — Payment Gateway
CRITICAL Confidence: 10/10

DeFOSPAM Finding: 'Missing Payment Gateway Decline/Failure Recovery'

Current Gap: No fallback if primary gateway fails; no redundant provider

Solution: Integrate 2+ payment providers (Stripe + PayPal); implement payment orchestration layer; automatic failover <10s

NFR-AVL-009: Distributed Locking for Stock Race Condition
CRITICAL Confidence: 10/10

DeFOSPAM Finding: 'Race Condition in Stock Validation at Order Confirmation'

Problem: Two customers simultaneously validate inventory, both see product in stock, both complete checkout, but stock depletes only once (overselling)

Solution: Pessimistic locking (SELECT...FOR UPDATE); distributed locks (Redis); timeout 1 second

NFR-AVL-006: RPO — Zero Data Loss for Orders/Payments
CRITICAL Confidence: 10/10

Requirement: Orders and payment transactions are financial records; losing even one transaction violates audit and compliance

Implementation: Synchronous database replication (both primary and secondary acknowledge write); replication lag <100ms

5 Critical Bottlenecks

NFR-SCL-002: Database Connection Pool Exhaustion at 5K Users
CRITICAL Confidence: 10/10

Current Pool: 200 connections (120 app, 50 web, 30 admin)

At 5K Peak Users: 880 queued connections waiting for pool slot; queue wait time spikes to 500+ ms

Solution: Deploy PgBouncer connection pooling middleware; increase pool from 200 → 300-400; transaction-mode multiplexing

NFR-SCL-003: Race Condition in Stock Validation
CRITICAL Confidence: 10/10

At Peak 2,500 Users: 250 customers may simultaneously add same product to cart

Without Locking: Inventory can go negative; 10-50% overselling possible for flash sale items

Test Required: Spike test with 100 concurrent orders for 5-unit item; verify no overselling

NFR-SCL-004: Payment Gateway Cascade Failure
CRITICAL Confidence: 10/10

Risk: Payment gateway 5-second response time blocks all checkout threads

Impact: Thread pool exhaustion → platform-wide outage (even non-checkout users affected)

Solution: Circuit breaker pattern; fail-fast on timeout; async payment queue with retry

NFR-SCL-005: Session Management Architecture Undefined
CRITICAL Confidence: 10/10

Current: Requirements assume Redis but don't mandate it

Risk: If sessions stored in local app memory → sticky sessions required → load balancing breaks → scaling fails

Requirement: ALL sessions MUST go to Redis cluster; no local session storage

Automated Mitigation Triggers

NFR-MON-009: 8 Automated Mitigation Actions
MAJOR Confidence: 8/10
  1. Auto-scale app tier: CPU >85% for 2 min → spin up 2 servers
  2. Circuit breaker for payment: Response >5s or error >5% → fail-fast, queue for retry
  3. Connection pool recovery: Utilization >80% → drain idle connections
  4. Cache eviction recovery: Eviction rate >100/sec → reduce TTLs by 50% for 30 min
  5. Traffic shedding: Network saturation >80% → shed non-critical features (recommendations, wishlists)
  6. Search slowness: p95 >1200ms + high GC → flush old cache, reduce facet aggregation
  7. Memory leak: Heap growth >100MB/hour → trigger heap dump, consider rolling restart
  8. Rate limit activation: Login error rate >0.5% (brute-force) → activate aggressive rate limiting
