Scalability Assessment

Critical Bottlenecks & Breaking Points

BOTTLENECK-1
Critical
Booking Correlation Query (Highest Risk)
Bottleneck: Scanning 15M passenger bookings per disruption event
Current Capacity: 50 concurrent disruptions
Target Capacity: 200 concurrent disruptions
Breaking Point: 150 concurrent disruptions (latency: 30+ seconds, exceeds SLA)

Mitigation: Partition bookings by route, implement bloom filter, add read-replica cluster (6+ nodes).

BOTTLENECK-2
Major
External Provider Rate Limits
Bottleneck: SMS 100 msg/sec, Email 50 msg/sec (150 msg/sec combined)
Current Capacity: 5,400 notifications/hour
Target Capacity: 30,000 notifications/hour
Breaking Point: 6,000 notifications/hour

Mitigation: Leaky bucket rate limiter, exponential backoff, batch emails (10x reduction), dual-provider fallover.

BOTTLENECK-3
Critical
Idempotency Key Cache Scaling
Bottleneck: Redis lookup on (Case_ID + channel + template_version)
Current Capacity: 1,000 dedup requests/sec
Target Capacity: 50,000 dedup requests/sec
Breaking Point: 500 dedup requests/sec

Mitigation: Redis Cluster with 6+ shards, in-process write-through cache (1min TTL).

BOTTLENECK-4
Major
Timeline Audit Log Growth
Bottleneck: Append-only log for 7-year retention (735M+ entries projected)
Current Write Rate: 0.56 entries/sec (normal), 6 entries/sec (peak)
Risk: B-tree index fragmentation, sequential scan degradation

Mitigation: Partition by (case_id, date), LSM tree storage, quarterly archive to S3 (Glacier).

Key Findings by Discipline

C Capacity Findings

NFR-CAP-001
Critical
Database Sizing for Booking Correlation
15M bookings must be indexed for sub-100ms lookups. 4 DB servers insufficient for 50+ concurrent disruptions.
NFR-CAP-002
Critical
Immutable Event Storage for 7-Year Retention
50K events/year * 5KB each = 0.25GB/year. 7-year total ~1.75GB. Append-only design means no compaction possible.
NFR-CAP-003
Critical
Notification Throughput for Peak Load
Peak: 30 disruptions * 150 passengers * 2 channels = 1000 notifications/hour. Requires async queuing with rate limiting.

L Latency Findings

NFR-LAT-001
Critical
End-to-End SLA: 20 Seconds (P95)
Formal SLA commitment required. Measure from ingestion received_timestamp to provider delivery_confirmation. Alert at p95 >= 18s.
NFR-LAT-004
Critical
Correlation Query Bottleneck
Booking query on 50M records with (service_no, date, O/D) matching is critical path. Must implement composite index, read replicas, result set limits.
NFR-LAT-006
Critical
Consent Real-Time Evaluation
GDPR requires consent checked at dispatch time, not snapshot at Case creation. Implement Redis cache (TTL 60s) with DB fallback.

A Availability Findings

NFR-AVL-001
Critical
Orchestrator Failure Recovery
DeFOSPAM flagged: orchestrator crash leaves events in intermediate state. Implement heartbeat (30s), pending event queue (5min TTL), state reconciliation.
NFR-AVL-002
Critical
Provider Retry Strategy
Twilio/SendGrid failures must be retried with exponential backoff (1s, 2s, 4s, 8s, max 5 attempts). Circuit breaker failover to secondary provider.
NFR-AVL-011
Critical
System Availability Target: 99.95%
EU Regulation 261/2004 requires timely passenger notification. 99.95% permits 262 minutes unplanned downtime annually (planned maintenance excluded).

S Scalability Findings

NFR-SCL-001
Critical
Booking Query Scaling
Current capacity: 50 concurrent disruptions. Target: 200. Mitigation: partition by route, bloom filters, read replicas (6+ nodes).
NFR-SCL-002
Critical
Provider Rate Limit Bottleneck
SMS 100/sec, Email 50/sec = 150 msg/sec hard limit. Current capacity 5,400/hr insufficient for 10x events. Use queue backpressure, leaky bucket.

S Security Findings

NFR-SEC-001
Critical
JWT Deep Links with Revocation
Deep links vulnerable to enumeration attacks without authentication. Implement RS256 JWT with 120min TTL, revocation blacklist in Redis.
NFR-SEC-002
Critical
Real-Time GDPR Consent Evaluation
Consent must be evaluated at dispatch, not Case creation. Real-time check violates SLA unless cached. Use Redis (60s TTL) with DB fallback.
NFR-SEC-005
Critical
PII Encryption
Passenger names, phone, email must be encrypted in transit (TLS 1.3) and at rest (AES-256). GDPR Articles 32-34 compliance required.

U Usability Findings

NFR-USE-001
Critical
SMS Clarity Within 5 Seconds
SMS must convey disruption type, impact, new time, booking ref in 160 chars max. Readability target: Flesch Reading Ease > 60.
NFR-USE-002
Critical
Deep Link Page Load on 4G
Page must load TTI < 3s on 4G. Passengers at airport with poor connectivity need fast access to case details. Page size < 50KB.

E Endurance Findings

NFR-END-001
Critical
Event Storage Growth
7-year immutable event log grows unbounded. Monitor: table size, append latency, disk I/O. Archive strategy required for operational queries.
NFR-END-002
Critical
Idempotency Cache Eviction
10M+ cache entries projected over 2 years. Without LRU eviction, cache consumes 2GB+ RAM. Implement explicit TTL (24 hours).
