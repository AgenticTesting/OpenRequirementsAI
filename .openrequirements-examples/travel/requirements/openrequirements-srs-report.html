1. Introduction

1.1 Purpose

STATUS: MISSING (TBD-SEC-001)
System purpose statement required to articulate:
- Airline operational challenge: Real-time distribution of disruption information to passengers
- System capability: Event detection, correlation, notification dispatch
- Business value: Passenger satisfaction, reduced call center load, operational transparency

1.2 Scope

In Scope:

  • Ingests disruption signals from carrier operations feeds (delays, cancellations, aircraft changes)
  • Stores raw events immutably with full metadata for audit trails (7-year retention)
  • Normalizes heterogeneous disruption formats into standardized Canonical Event Schema
  • Correlates disruptions to passenger bookings using service_no+date+origin/destination matching
  • Creates Case records with versioned severity computation
  • Evaluates real-time passenger consent before dispatch (GDPR Article 7 compliance)
  • Composes contextual SMS/Email/Push notifications
  • Prevents duplicate notifications via idempotency keys (exactly-once guarantee)
  • Enforces passenger quiet hours with service-critical exemptions
  • Maintains immutable append-only timeline for every Case lifecycle event

Out of Scope:

  • Journey planning or itinerary rebooking
  • Refund or compensation processing
  • Real-time flight status prediction
  • Manual case management workflows

2. Overall Description

2.1 Product Perspective

4 External System Interfaces:

  1. Carrier Operations Feed: Multiple protocols (ACARS, IATA SSIM, airline APIs, manual entry)
  2. Notification Providers: Current 5,400 msg/hour; target 30,000 msg/hour (v1.1)
  3. Consent Authority: Real-time query with p95 ≤ 1,200ms latency
  4. Deep Link Service: JWT-based tokens with 120-minute TTL
2.2 Product Functions

10 Core Features (v1.0 MVP):

1. Disruption Event Ingestion and Storage - Ingest raw signals with metadata preservation (7-year retention, 11.15GB current)
2. Event Normalization - Transform to Canonical Event Schema (p95 500ms)
3. Disruption-to-Booking Correlation - Deterministic matching against 50M booking records (p95 4,000ms)
4. Case Management and Severity - Create Cases with versioned decision tree (p95 700ms)
5. Passenger Consent and Channel Selection - GDPR-compliant real-time consent verification (p95 1,200ms)
6. Notification Composition - SMS with 3 elements, max 160 chars, Flesch Reading Ease ≥ 60
7. Template-Based Dispatch - Versioned locale-specific templates with automatic retry (p95 5,000ms)
8. Idempotent Deduplication - SHA256-based cache with 24-hour TTL (50,000 lookups/sec)
9. Quiet Hours and Exemptions - Locale-aware timezone-aware enforcement with CRITICAL bypass
10. Timeline Auditing - Immutable append-only 8-event lifecycle tracking
2.3 User Characteristics

5 User Types: Passengers (novice, SMS-only), Airline Operations (intermediate), Consent Authority Admin (privacy), SRE/Monitoring (advanced), Audit Investigator (compliance)

2.4 Constraints

System: 99.95% availability (262 min annual downtime), RTO 5 min, RPO 0
Data: 7-year retention, 11.15GB current, 28.3GB projected 24-month
Performance: p95 ≤ 20-second end-to-end SLA with component budgets
Security: TLS 1.3, AES-256 at-rest, GDPR Article 7, JWT RS256

2.5 Assumptions and Dependencies

Key Assumptions: Deterministic input data, real-time consent available, SMS provider minimum capacity, booking database queryable, immutable storage technology available

Unvalidated Assumptions (Risk): 5-15% disruption rate, 20% YoY growth, 7-year retention source, 90% consent cache hit rate, 50M booking database size
